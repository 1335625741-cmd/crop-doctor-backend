# Output Schema(农户版回复结构)

模型返回 JSON 后,按这个 schema 整理成给农户看的回复。
**核心原则:**
1. **开头必含免责声明**(固定模板,一字不差)
2. **多采用表格**,减少大段文字
3. **emoji 克制使用**:仅保留必要场景(严重程度色块 / 置信度星标 / 候选奖牌 / 安全警告 / 打药时间 / 复喷);去掉装饰性 emoji(段落标题、方案类型、反馈按钮、类别、顶部标签)
4. **多候选分类列方案**:每个候选诊断独立成块,命中处方的给完整药剂表,没命中的给"农业措施 + 鉴别建议"表

---

## ⚠️ 0. 免责声明(必含,放在回复**最开头**)

固定模板,**一字不差**使用:

> ⚠️ **免责声明**:本回复由 AI 辅助诊断,仅作技术参考。**严禁仅凭本回复用药或处置**。实际用药请严格遵守:**① 当地农资店**的库存与建议、**② 当地农技员**的现场指导、**③ 药剂标签**的剂量和安全间隔期。结合具体气候、作物、生育期综合判断,如有疑问请联系当地植保站或农业部门。

(任何时候都不能省略这一段。删掉就要担责。)

## 📋 1. 诊断概要(单行小表格,2-3 行)

| 字段 | 值 |
|---|---|
| **病名** | <Top 1 name> |
| **概率** | **<XX.XX%>**(见下方计算方法) |
| **类别** | <category 转白话>(**不加类别 emoji**) |
| **严重程度** | <severity + 色块 emoji> |
| **大致原因** | <cause_summary,白话> |

**严重程度色块 emoji 映射**:

| severity | emoji |
|---|---|
| 轻 | 🟢 |
| 中 | 🟡 |
| 重 | 🟠 |
| 无法判断 | ⚪ |

## 🔍 2. 候选诊断详情(单表,所有候选一行一个)

| 候选 | 病名 | 类别 | 病原 | 概率 | 置信度 | 关键视觉特征 |
|---|---|---|---|---|---|---|
| 🥇 Top 1 | <name> | <category> | <pathogen> | **<XX.XX%>** | <置信 emoji> <confidence> | <clue 1>; <clue 2> |
| 🥈 Top 2 | <name> | <category> | <pathogen> | <XX.XX%> | <置信 emoji> <confidence> | <clue 1> |
| 🥉 Top 3 | <name> | <category> | <pathogen> | <XX.XX%> | <置信 emoji> <confidence> | <clue 1> |

**置信度星标映射**(只用星,不用文字堆叠):

| confidence | emoji |
|---|---|
| 高 | ⭐⭐⭐ |
| 中 | ⭐⭐ |
| 低 | ⭐ |

**候选奖牌 emoji**:🥇 Top 1 / 🥈 Top 2 / 🥉 Top 3

**概率字段**(必须展示):

| 字段 | 类型 | 取值范围 | 显示格式 |
|---|---|---|---|
| `probability` | 浮点 | 0.00-1.00(归一化后总和=1.00) | **XX.XX%**(百分比,小数点后 2 位,加粗) |

**概率计算方法**:

1. **LLM 自评(主)**:视觉模型基于图像特征 + 用户上下文(作物/天气/部位/用药史)给出每个候选的原始概率数字
2. **Fallback**:如 LLM 未给具体数字,按 confidence 字段映射(高 → 0.85 / 中 → 0.65 / 低 → 0.35)
3. **候选间 softmax 归一化**:所有候选的原始概率做 softmax,放大 Top 1 优势,确保总和 = 1.00
4. **显示**:`probability × 100`,保留 2 位小数,加 %

**LLM 自评说明**:LLM 的"概率"是**主观打分**(非严格数学概率),反映模型对诊断的确信程度。实际处理时仍要结合农资店判断和现场观察,不能把 85.00% 当成数学概率理解。

**为每个候选诊断独立成块**,命中处方的给完整药剂表,没命中的走降级格式。


## 🎨 emoji 使用原则总结

| 类别 | 是否用 emoji | 理由 |
|---|---|---|
| ⚠️ 免责声明 / 安全警告 | ✅ 用 | 必要警告,不能省 |
| 🟢🟡🟠⚪ 严重程度色块 | ✅ 用 | 视觉一眼识别 |
| ⭐ 置信度星标 | ✅ 用 | 视觉一眼识别 |
| 🥇🥈🥉 候选奖牌 | ✅ 用 | 视觉一眼识别 |
| ⏰ 打药时间 | ✅ 用 | 辅助理解,小图标 |
| 🔁 复喷 | ✅ 用 | 辅助理解,小图标 |
| 段落标题(📋🔍✋💊🆘🔎❓💬) | ❌ 不用 | 装饰性,会让回复像 emoji 大杂烩 |
| 方案类型(🛡️🎯🧪🌱💧) | ❌ 不用 | 装饰性,文字"保护性/治疗性"够清楚 |
| 反馈按钮(✅👍😐😟⏳) | ❌ 不用 | 装饰性,纯字母 A/B/C/D/E 更简洁 |
| 类别(🍄🐛🧪⚠️🌡️) | ❌ 不用 | 装饰性,文字"病害/虫害/缺素"够清楚 |
| 顶部标签(🌱) | ❌ 不用 | 装饰性,纯文字"作物病害诊断报告"够 |

## 解析失败时的兜底

如果模型返回的不是合法 JSON:

1. 尝试用 ```json ... ``` 标记剥取
2. 还是不行就告诉用户"模型这回返回的格式我没解析出来,先把它给的建议转给你看"
3. 把模型原始文本按上面模板重新组织(白话化 + 表格化)
4. 仍然遵循"开头免责声明 + 表格优先 + emoji 克制"原则
5. 不要丢弃模型给的信息,降级展示

**字段说明**:
- `query.description`: GLM-4V 对用户上传图的一句话描述(1-2 句中文)
- `rag_references`: RAG top-K 候选(默认 3,`--top-k` 可调),按 `score` 降序
- `diagnose.diagnosis[]`: 二轮 LLM 综合判断的候选诊断(跟原 schema 一致,但多了 `reasoning` 字段引用 RAG 候选)
- `diagnose.rag_used`: 评估 RAG 准确率,`agreed_with_top1=true` 表示 LLM 最终判断跟 RAG top-1 一致

**渲染时怎么用**:
1. `query.description` 作为"用户问诊背景",给 LLM 看的
2. `rag_references[]` 展示给用户看("我们从本地历史图库找到的最相似的 3 张,你看是不是这个病?")
3. `diagnose.diagnosis[0]` 作为"最终诊断"输出到农户版回复(走第 1-8 章节的现有模板)

- `full-diagnosis-with-rag.json` 与 `full-diagnosis.json` 完全兼容
